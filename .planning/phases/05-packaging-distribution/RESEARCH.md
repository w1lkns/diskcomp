## Summary

This phase requires establishing two parallel installation paths (pip + single-file) with a working CI matrix and complete package metadata. The research validates that:

1. **Hatchling is the right build backend** for stdlib-only packages — simpler than setuptools, well-maintained, requires only `[build-system]` + `[project]` tables
2. **GitHub Actions matrix syntax is straightforward** — v5 checkout/setup-python actions, exclude combinations not needed, run full test suite per matrix combo
3. **Module bundling is manual but safe** — read modules in dependency order, strip internal imports, combine stdlib imports, preserve `if __name__ == "__main__"` guards to prevent entry point pollution
4. **shields.io badges work with exact workflow filenames** — GitHub Actions badge uses workflow YAML filename (e.g., `ci.yml`), PyPI version badge works even before publish with `/pypi/v/packagename`
5. **`#!/usr/bin/env python3` shebang is fully portable** — Windows Python Launcher (v3.3+) natively interprets it; modern Windows 10+ with Terminal/PowerShell supports ANSI codes

**Primary recommendation:** Use Hatchling + pyproject.toml with `requires-python = ">=3.8"`, GitHub Actions v5 matrix across 9 combos (3 OS × 3 Python versions), and a `build_single.py` script that concatenates modules in dependency order while stripping relative imports and preserving module-level guards.

### Pattern 1: Build System Configuration (pyproject.toml)
**What:** Declarative metadata + build backend configuration per PEP 621 and PEP 518

**When to use:** For all modern Python packages (2021+); required by pip, setuptools, hatch, poetry

**Example (minimal, production-ready):**
```toml
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "diskcomp"
version = "0.1.0"
description = "Compare two drives and find duplicate files"
authors = [
    { name = "Wilkins Morales", email = "wilkins@example.com" }
]
readme = "README.md"
requires-python = ">=3.8"
license = { text = "MIT" }
classifiers = [
    "Development Status :: 3 - Alpha",
    "Intended Audience :: End Users/Desktop",
    "License :: OSI Approved :: MIT License",
    "Operating System :: OS Independent",
    "Programming Language :: Python :: 3",
    "Programming Language :: Python :: 3.8",
    "Programming Language :: Python :: 3.10",
    "Programming Language :: Python :: 3.12",
    "Topic :: System :: Filesystems",
    "Topic :: Utilities",
]
keywords = ["disk", "duplicate", "files", "storage"]

[project.urls]
Repository = "https://github.com/w1lkns/diskcomp"
Issues = "https://github.com/w1lkns/diskcomp/issues"

[project.optional-dependencies]

[project.scripts]

```

**Key points:**
- `requires-python = ">=3.8"` restricts installation; classifiers are documentation only
- `[project.scripts]` creates entry point: `diskcomp` command → `diskcomp.cli:main` function
- `[project.optional-dependencies]` allows `pip install diskcomp[rich]`; no version pin on rich needed here (graceful fallback)
- No setup.py needed; hatchling auto-detects package structure
- License field: can use `{text = "MIT"}` or `{file = "LICENSE"}` if LICENSE file exists

### Pattern 2: Single-File Bundling (build_single.py)
**What:** A dev-time script that reads all modules in dependency order, strips internal imports, and writes `diskcomp.py`

**When to use:** For projects requiring zero-dependency distribution (not pip install) or offline usage

**Module dependency order (from CONTEXT.md + code inspection):**
```
1. types.py              → no internal deps
2. ansi_codes.py        → no internal deps
3. scanner.py           → imports: types
4. hasher.py            → imports: types
5. reporter.py          → imports: types
6. benchmarker.py       → no internal deps
7. health.py            → imports: types, benchmarker
8. drive_picker.py      → imports: types, health
9. ui.py                → imports: ansi_codes (+ optional rich)
10. deletion.py         → imports: types
11. cli.py              → imports: all above
```

**Algorithm (pseudocode):**
```
1. Read all files in order above
2. For each file:
   a. Extract all stdlib imports (os, sys, json, etc.)
   b. Skip lines matching: from diskcomp.xxx import / import diskcomp.xxx
   c. Keep all class, def, CONSTANT definitions
   d. Keep `if __name__ == "__main__":` guards (don't strip — needed for entry point safety)
3. Combine stdlib imports at top (deduplicate)
4. Add shebang: #!/usr/bin/env python3
5. Write to diskcomp.py with header comment
```

**Example (snippet from build_single.py):**
```python
#!/usr/bin/env python3
"""
Auto-generates diskcomp.py from diskcomp/ package modules.
Run this after any changes to package source code.
"""

import os
import re

MODULES_IN_ORDER = [
    "types",
    "ansi_codes",
    "scanner",
    "hasher",
    "reporter",
    "benchmarker",
    "health",
    "drive_picker",
    "ui",
    "deletion",
    "cli",
]

def extract_code(filepath):
    """Read file, skip diskcomp imports, keep stdlib + code."""
    with open(filepath) as f:
        lines = f.readlines()

    stdlib_imports = set()
    code_lines = []
    in_docstring = False

    for line in lines:
        # Skip diskcomp package imports
        if re.match(r'^from diskcomp\.|^import diskcomp', line):
            continue

        # Collect stdlib imports
        if line.startswith(('import ', 'from ')) and 'diskcomp' not in line:
            stdlib_imports.add(line.rstrip())
        else:
            code_lines.append(line)

    return stdlib_imports, code_lines

def main():
    """Concatenate modules and write diskcomp.py"""
    all_stdlib = set()
    all_code = []

    for module in MODULES_IN_ORDER:
        filepath = f"diskcomp/{module}.py"
        stdlib, code = extract_code(filepath)
        all_stdlib.update(stdlib)
        # Skip module docstring and blank lines from bundled modules
        all_code.extend(code)

    with open("diskcomp.py", "w") as f:
        f.write("#!/usr/bin/env python3\n")
        f.write("# Auto-generated by build_single.py — do not edit directly\n\n")

        # Write deduplicated stdlib imports
        for imp in sorted(all_stdlib):
            f.write(imp + "\n")
        f.write("\n\n")

        # Write all code
        f.write("".join(all_code))

if __name__ == "__main__":
    main()
```

**Key gotchas to avoid:**
1. **Relative imports:** Strip `from diskcomp.xxx import` — absolute imports only in bundled file
2. **`if __name__ == "__main__":` guards:** Keep these! They prevent code running when `diskcomp.py` is imported as a module (edge case but important)
3. **Module docstrings:** Strip leading docstring from each bundled module (keep only code definitions)
4. **Duplicate imports:** Deduplicate stdlib imports at top (use a set)
5. **Try/except for optional imports:** Keep these intact (Rich imports in ui.py are wrapped; preserve them)

### Pattern 3: GitHub Actions Matrix CI (ci.yml)
**What:** YAML workflow that tests code against multiple OS + Python version combos in parallel

**When to use:** For any library targeting multiple Python versions (3.8, 3.10, 3.12) and platforms

**Example (production-ready):**
```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
        python-version: ["3.8", "3.10", "3.12"]

    steps:
      - uses: actions/checkout@v4

      - name: Set up Python ${{ matrix.python-version }}
        uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
          cache: pip

      - name: Install package (editable)
        run: pip install -e .

      - name: Run tests
        run: python -m unittest discover tests/

      - name: Test single-file build
        run: python build_single.py && python diskcomp.py --help
```

**Key points:**
- `fail-fast: false` — continue all tests even if one combo fails (useful for debugging)
- `cache: pip` — caches pip dependencies between runs (~30s speedup)
- `actions/checkout@v4` — modern stable version; v5 is available but v4 widely used
- `actions/setup-python@v5` — handles Python installation across all OSes
- Two test steps: (1) test installed package, (2) test single-file generation
- No secrets/environment variables needed (CI runs on public code)

**Matrix explanation:**
- **9 jobs total:** 3 OS × 3 Python versions
- **Each job:** checks out code → installs Python → installs package → runs tests
- **Result:** ensures code works on all combos before merge

### Anti-Patterns to Avoid
- **Including `setup.py` when using hatchling:** Redundant, confusing — hatchling auto-detects package; setup.py only needed for C extensions
- **Manual version numbers in pyproject.toml + code:** Should be single source of truth (currently `diskcomp/__init__.py` has `__version__ = "0.1.0"`); can be referenced in pyproject.toml with `dynamic = ["version"]` + version config
- **Relative imports in single-file bundle:** Will fail at runtime when `diskcomp.py` is a top-level script; always strip and rewrite as absolute
- **Checking in `diskcomp.py` to git:** Generated artifact; add to `.gitignore`
- **Testing only on one Python version in CI:** Matrix design ensures you catch version-specific bugs (e.g., typing changes between 3.8→3.12)

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Build system for Python package | setup.py with setuptools boilerplate | pyproject.toml + hatchling | Modern standard; setuptools ~200 LOC vs hatchling ~20 lines; PEP 517/518/621 standardized |
| Package entry point (CLI) | Manual script wrapper | `[project.scripts]` in pyproject.toml | pip handles creation of `diskcomp` command automatically; setuptools/hatchling wire argparse → sys.exit() correctly |
| CI matrix across OS/Python versions | Manual separate jobs | GitHub Actions `strategy.matrix` | Matrix syntax creates N jobs automatically; maintains consistency; easier to extend |
| Single-file module concatenation | Manual file append | Structured `build_single.py` script | Naive concatenation breaks imports, duplicate code, loses docstrings; proper bundler handles dependency order + import stripping |
| Badge URLs in README | Hard-coded shield URLs | shields.io templated URLs | shields.io URLs auto-update (e.g., CI status re-fetched every build); hard-coded would be stale |

**Key insight:** The Python packaging ecosystem standardized around PEP 517/518/621 (2017-2021) specifically to eliminate boilerplate and hand-rolled solutions. Using pyproject.toml + hatchling means you don't maintain build config if you switch ecosystems later.

## Common Pitfalls

### Pitfall 1: Confusing Module Import Stripping in build_single.py
**What goes wrong:** Bundler strips `from diskcomp.scanner import FileScanner` but leaves the original relative reference in bundled code, causing NameError at runtime when `diskcomp.py` runs as top-level script.

**Why it happens:** build_single.py must rewrite internal imports to become bare references (since bundled code no longer has package structure). Naively stripping lines doesn't rewrite the reference.

**How to avoid:** After stripping `from diskcomp.xxx import YYY`, ensure bundled code uses YYY (not qualified name), since all classes/functions will be in global scope of bundled file.

**Warning signs:** `NameError: name 'FileScanner' is not defined` when running `python diskcomp.py`, even though class is defined in file.

### Pitfall 2: Forgetting if __name__ == "__main__": Guard in Bundle
**What goes wrong:** If cli.py has top-level code like `args = parse_args()` outside the `main()` function, bundling it executes that code during import, breaking the entry point.

**Why it happens:** When diskcomp.py is used as a module (rare but possible), module-level code runs at import time, not at CLI invocation.

**How to avoid:** Ensure all entry-point logic is inside `if __name__ == "__main__":` guard (already true in diskcomp/cli.py). build_single.py must NOT strip these guards.

**Warning signs:** `diskcomp` command hangs or runs without arguments; entry point tries to prompt for input during module load.

### Pitfall 3: Build Backend Not Found in CI
**What goes wrong:** CI runs `pip install -e .` but hatchling is not installed, failing with `ERROR: No module named 'hatchling'`.

**Why it happens:** GitHub Actions runs fresh Python installations; `[build-system] requires = ["hatchling"]` must be present in pyproject.toml for pip to auto-install build backend.

**How to avoid:** Always include `[build-system]` section in pyproject.toml. pip reads this and installs backend before building.

**Warning signs:** CI log shows `error: invalid command 'bdist_wheel'` or module not found; local `pip install -e .` works (because you already have hatchling globally).

### Pitfall 4: Windows Shebang Issues
**What goes wrong:** Single-file `diskcomp.py` with `#!/usr/bin/env python3` fails to execute on Windows; user tries `./diskcomp.py` but shell doesn't recognize shebang.

**Why it happens:** Windows doesn't support shebangs at kernel level (Unix feature). Users must use `python3 diskcomp.py`, not `./diskcomp.py`.

**How to avoid:** Document both invocations in README: `python3 diskcomp.py` (all platforms, always works) and `./diskcomp.py` (macOS/Linux only, requires chmod +x). Python Launcher for Windows (v3.3+) does recognize shebangs in Terminal, but command-line interpretation varies.

**Warning signs:** Works on macOS/Linux CI, fails on Windows CI; user reports "file not found" on Windows.

### Pitfall 5: Missing Classifiers Breaking PyPI Search
**What goes wrong:** pyproject.toml omits Python version classifiers (e.g., `"Programming Language :: Python :: 3.8"`), so package doesn't appear in PyPI search for "Python 3.8" even though `requires-python = ">=3.8"` is set.

**Why it happens:** PyPI search uses classifiers, not `requires-python` (which only blocks installation). These are two separate metadata fields.

**How to avoid:** Include classifiers for every tested Python version. See Standard Stack table for exact strings.

**Warning signs:** Package installs fine on Python 3.8, but doesn't appear in PyPI search filters; users can't find it by Python version.

### Pitfall 6: Optional Dependency Import Failures
**What goes wrong:** User installs `pip install diskcomp` (without `[rich]`), then code tries `from rich.progress import Progress` without try/except, raising ImportError.

**Why it happens:** ui.py imports Rich optionally with try/except, but if someone adds a new module that imports Rich directly (no guard), bundle breaks.

**How to avoid:** Code review: all Rich imports must be wrapped in try/except. Set RICH_AVAILABLE flag, use it in factories (UIHandler.create() checks RICH_AVAILABLE). Maintain this invariant.

**Warning signs:** `ImportError: No module named 'rich'` when running diskcomp without installing [rich] extra.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| distutils / setup.py | pyproject.toml + build backend | ~2021 (PEP 517/518/621) | Removed 200+ lines of boilerplate; build backend can be swapped without touching source |
| Manual setup.cfg | Declarative [project] table in pyproject.toml | ~2021 | Single file for all metadata; integrates with all tools (pip, hatch, poetry, setuptools) |

**Deprecated/outdated:**
- **setup.py with setuptools:** Modern pip/hatch/poetry don't require it; use pyproject.toml exclusively. (Still used for projects with C extensions, custom build steps — not needed here.)
- **pkg_resources for entry points:** `pkg_resources.iter_entry_points()` is slow and implicit; [project.scripts] in pyproject.toml is explicit and faster.
