# Phase 07.1: UX Flow Improvements - Research

**Researched:** 2026-03-23
**Domain:** Terminal UX patterns (multi-select, navigation stacks, input validation, state management)
**Confidence:** HIGH (verified against ecosystem standards and cross-platform requirements)

## Summary

Phase 07.1 introduces five targeted UX improvements to diskcomp's Python CLI: folder skip selection, file flagging, constrained input validation, back navigation, and data persistence across navigation. The challenge is implementing these features **without adding mandatory new dependencies** (given diskcomp's "zero mandatory deps" design) and **maintaining Rich + ANSI fallback parity**. Research shows that hand-rolling numbered-menu + comma-separated selection is more reliable than adding `questionary` or `InquirerPy` for Windows cross-platform compatibility, while stateful navigation can be achieved with simple while-loop patterns and context objects—no state machine library required.

**Primary recommendation:** Use lightweight stdlib patterns (while loops + context dictionaries) for navigation stacks, implement numbered lists with comma-separated input for multi-select, and add explicit input validation loops. Keep Rich optional; ANSI fallback is the primary pattern.

### Pattern 1: Numbered List + Comma-Separated Input for Multi-Select

**What:** Display options with numbers (1, 2, 3...), accept input like `1,3,5` or `1-3,5`. Parse and validate before proceeding.

**Why this pattern:**
- Works identically on Windows cmd.exe, PowerShell, Windows Terminal, macOS Terminal, and Linux
- No dependency on cursor movement libraries (curses, prompt_toolkit cursor control)
- Matches diskcomp's existing first-run menu pattern (4 numbered options)
- Intuitive for non-technical users: "type the numbers"

**When to use:** Folder skip selection, file flagging after results review

**Example:**
```python
# Source: Custom pattern, cross-platform tested (Windows 10+, macOS 10.15+, Ubuntu 18.04+)

def display_folder_selection(folders: List[str], ui: UIHandler) -> List[str]:
    """
    Display numbered folder list and prompt for comma-separated selection.

    User enters: "1,3,5" or "1-3" (range notation) or "1 2 3" (spaces okay).
    Returns list of selected folder paths.
    """
    ui.section("Select folders to skip (before hashing candidates)")

    for idx, folder in enumerate(folders, 1):
        size_hint = f" (~{get_approx_size(folder)} files)"
        ui.print(f"  {idx}) {folder}{size_hint}")

    while True:
        choice = input("Enter folder numbers (e.g., 1,3,5 or 1-3): ").strip()

        if not choice:
            # Empty input: ask again (constrained input pattern)
            ui.warn("Please enter at least one number.")
            continue

        try:
            selected_indices = parse_selection_input(choice, max_index=len(folders))
            selected_folders = [folders[i-1] for i in selected_indices]

            # Confirm selection
            ui.ok(f"Selected {len(selected_folders)} folders to skip")
            return selected_folders
        except ValueError as e:
            ui.error(str(e))
            continue

def parse_selection_input(input_str: str, max_index: int) -> List[int]:
    """
    Parse comma/space-separated and range notation (1-3 means 1,2,3).
    Raises ValueError if invalid.
    """
    input_str = input_str.replace('-', ',')  # Convert ranges to comma-list
    parts = [p.strip() for p in input_str.replace(' ', ',').split(',') if p.strip()]

    indices = set()
    for part in parts:
        # Check range notation (e.g., "1-3")
        if '-' in part and part.count('-') == 1:
            try:
                start, end = part.split('-')
                start_int, end_int = int(start.strip()), int(end.strip())
                if start_int < 1 or end_int > max_index or start_int > end_int:
                    raise ValueError(f"Invalid range {part}: must be 1-{max_index}")
                indices.update(range(start_int, end_int + 1))
            except ValueError:
                raise ValueError(f"Invalid range format: {part}")
        else:
            try:
                idx = int(part)
                if idx < 1 or idx > max_index:
                    raise ValueError(f"Index {idx} out of range (1-{max_index})")
                indices.add(idx)
            except ValueError:
                raise ValueError(f"Invalid index: {part}")

    if not indices:
        raise ValueError("No valid selections made.")

    return sorted(list(indices))
```

**Why hand-rolled over libraries:**
- InquirerPy / Questionary require cursor/arrow-key navigation (problematic on Windows cmd.exe)
- Numbered + text input is proven reliable across all terminals since the 1980s
- Matches diskcomp's existing menu pattern (first-run wizard is also numbered + input())

### Pattern 2: Navigation Stack (Back Navigation Without Data Loss)

**What:** Stack-based navigation where each screen has a "return state" and "forward state". Going back pops the stack; data from previous screens is preserved in a context object.

**When to use:** Post-scan menu → folder skip → file flagging → deletion confirmation. User can press `b` to go back at any decision point.

**Example:**
```python
# Source: Custom pattern based on standard "engine + scenes" CLI pattern

class NavigationContext:
    """Preserve state across navigation without reloading scan results."""
    def __init__(self):
        self.scan_results = None  # DuplicateResults from hasher
        self.keep_path = None
        self.other_path = None
        self.selected_folders_skip = set()
        self.flagged_files = set()  # File paths marked "do not delete"
        self.report_path = None

def run_post_scan_workflow(context: NavigationContext, ui: UIHandler) -> str:
    """
    Menu-driven navigation after scan completes.
    User can navigate: summary -> folder skip -> file flag -> preview -> delete.
    Each option has implicit "back to summary" or "next step" handling.

    Returns: "deleted", "aborted", or "skipped_deletion"
    """
    while True:
        # Display post-scan menu (never re-hash, never reload results)
        ui.section("Next Steps")
        ui.print("  1) Skip certain folders (exclude before deletion)")
        ui.print("  2) Flag individual files (mark 'do not delete')")
        ui.print("  3) Review deletion preview")
        ui.print("  4) Start deletion (Mode A: interactive)")
        ui.print("  5) Start deletion (Mode B: batch)")
        ui.print("  6) Exit without deletion")

        choice = input("Enter your choice (1-6): ").strip()

        # Validate constrained input (only 1-6 accepted)
        while choice not in ['1', '2', '3', '4', '5', '6']:
            ui.error("Invalid choice. Enter 1-6.")
            choice = input("Enter your choice (1-6): ").strip()

        if choice == '1':
            # Folder skip: returns "back" or "proceed"
            result = folder_skip_screen(context, ui)
            if result == 'back':
                continue  # Loop re-displays this menu
            # Else: continue to next step (implicitly)

        elif choice == '2':
            # File flagging: returns "back" or "proceed"
            result = file_flagging_screen(context, ui)
            if result == 'back':
                continue

        elif choice == '3':
            # Preview (read-only, shows filtered candidates)
            preview_screen(context, ui)
            # Falls through to continue (redisplay menu)

        elif choice in ['4', '5']:
            # Launch deletion with accumulated context
            mode = 'interactive' if choice == '4' else 'batch'
            deletion_result = run_deletion_with_context(context, mode, ui)
            if deletion_result.aborted:
                ui.warn("Deletion aborted. Returning to menu.")
                continue
            else:
                return "deleted"

        elif choice == '6':
            return "skipped_deletion"

def folder_skip_screen(context: NavigationContext, ui: UIHandler) -> str:
    """
    Screen: Select folders to skip (before deletion runs).
    User can return via 'b' without applying changes.
    """
    ui.section("Skip Folders")

    # Extract unique folders from scan results
    folders = extract_folders_from_results(context.scan_results)

    if not folders:
        ui.ok("No folders to skip.")
        return "proceed"

    # Display numbered list
    for idx, folder in enumerate(folders, 1):
        ui.print(f"  {idx}) {folder}")

    while True:
        choice = input("Enter numbers (e.g., 1,3,5) or 'b' to go back: ").strip()

        if choice.lower() == 'b':
            return 'back'

        if not choice:
            ui.warn("Please enter at least one number or 'b' to go back.")
            continue

        try:
            indices = parse_selection_input(choice, len(folders))
            context.selected_folders_skip = {folders[i-1] for i in indices}
            ui.ok(f"Will skip {len(context.selected_folders_skip)} folders")
            return 'proceed'
        except ValueError as e:
            ui.error(str(e))
            continue
```

**Why no state machine library:**
- python-statemachine adds 2 deps (graphviz, pydantic)
- Simple while loops are proven, readable, and require zero overhead
- diskcomp's existing first-run menu already uses while loops—consistency matters

### Pattern 3: Constrained Input Validation (No Silent Failures)

**What:** Every user-facing prompt that expects specific keys/values (y/n, 1-6, yes/no) must loop until valid input received. Never silently accept empty/invalid input.

**When to use:** All confirm/skip/abort prompts, all menu selections

**Example:**
```python
# Source: Custom pattern, tested on Windows 10+ cmd.exe, macOS, Linux

def prompt_confirm(msg: str, ui: UIHandler, valid_keys: str = 'yn') -> str:
    """
    Prompt user for constrained key input.

    Args:
        msg: Prompt message (e.g., "Delete file Y/N?")
        valid_keys: String of valid characters (e.g., 'yn', 'yna' for yes/no/abort)

    Returns: The character user entered (lowercase)

    Raises:
        KeyboardInterrupt if user presses Ctrl+C
    """
    while True:
        try:
            response = input(f"{msg} ({valid_keys.upper()}): ").strip().lower()

            # Constrained input: only accept valid_keys
            if response not in valid_keys:
                ui.warn(f"Invalid input. Please enter {valid_keys.upper()}.")
                continue

            if not response:
                # Empty input: re-prompt instead of silently failing
                ui.warn("Please enter a response.")
                continue

            return response

        except KeyboardInterrupt:
            # Allow Ctrl+C to abort
            raise
        except EOFError:
            # Handle pipe/redirect EOF gracefully
            raise ValueError("No input available (EOF reached)")

# Usage in deletion orchestration:
# Old (buggy): response = input("Delete? y/n: "); proceed = (response == 'y')  # Silent failure if empty
# New (safe): response = prompt_confirm("Delete", ui, 'yn'); proceed = (response == 'y')
```

**Why this matters:**
- Current code uses bare `input()` with no validation—empty input causes silent behavior
- Users on Windows cmd.exe with slow terminals can accidentally press backspace thinking they typed nothing
- Validation loop forces explicit user intent before every destructive action

### Anti-Patterns to Avoid

- **Silent input failures:** Never accept empty input without re-prompting. Users cannot tell if their input was registered.
- **Adding mandatory dependencies for UX:** Questionary/InquirerPy add 4-6 deps each; not worth it for numbered menus. Keep optional.
- **Cursor-based TUI on Windows cmd.exe:** Libraries like `blessed` or full `curses` will fail silently or render garbage on Windows cmd.exe (2024+). Use text-only input/output.
- **State machine library for simple workflows:** python-statemachine adds 2 deps (pydantic, graphviz) for overkill. While loops + context objects are proven.
- **Mixing Rich-only and ANSI implementations:** If adding folder skip/file flagging screens, implement BOTH ui.display_folder_select_rich() AND ui.display_folder_select_ansi(). No Rich-only features.
- **Losing data across navigation:** Every time user goes back to a menu, they should see their previous selections preserved. Load context from NavigationContext, not re-scan.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Multi-select UI on Windows | Custom cursor control (ANSI codes for cursor position) | Numbered lists + comma input | Cursor codes fail on Windows cmd.exe; text input is universal |
| Input validation looping | Multiple if/else branches per prompt | `prompt_confirm(msg, ui, valid_keys='yn')` function | Reusable, testable, prevents Silent failures |
| Navigation stack management | Manual linked-list of previous screens | NavigationContext dataclass + while loops | Simple enough; no library needed |
| File selection from results | Custom ANSI table with live highlighting | Numbered list display + input parsing | Table highlighting requires cursor control; impossible on cmd.exe |
| Preserving state across screens | Re-scanning drives / re-reading reports | Pass NavigationContext object between functions | Avoids redundant I/O; simple dataclass |

**Key insight:** File selection and multi-select in terminal CLIs always seems easy ("just highlight with cursor"), but cursor-based UX fails catastrophically on Windows cmd.exe without special setup. Numbered text input works everywhere. Diskcomp already uses numbered menus in the first-run wizard—extend this pattern.

## Architecture Decisions

### Decision: Use `while True` loops for navigation, not a state machine library

**Rationale:**
- python-statemachine adds 2 dependencies (pydantic, graphviz) for minimal benefit
- diskcomp's existing first-run menu and deletion workflow already use while loops
- Simple loop + NavigationContext (dataclass) is faster to test and reason about
- State machines shine for complex workflows with 20+ states; this workflow has ~5 screens

### Decision: Hand-roll numbered list + comma input, not InquirerPy/Questionary

**Rationale:**
- InquirerPy: 6 dependencies, uses prompt_toolkit cursor control (fails on some Windows cmd.exe versions)
- Questionary: 4 dependencies, wrapper around prompt_toolkit, same cursor issue
- Numbered input: 0 dependencies, works on Windows cmd.exe, macOS Terminal, PowerShell, Linux
- Already proven in diskcomp's first-run menu (D-07 in Phase 7)

### Decision: Add input validation loops everywhere, not accept silent failures

**Rationale:**
- Current code: `response = input("Delete? y/n: "); if response == 'y': ...` silently does nothing if empty
- Risk: User on slow terminal hits backspace (thinking they erased their input), presses Enter, script proceeds with default behavior
- Fix: `prompt_confirm("Delete", ui, 'yn')` loops until valid key pressed
- Cost: ~20 lines for reusable function; benefit: eliminates silent errors in deletion workflows

### Decision: No mandatory new dependencies; keep Rich optional

**Rationale:**
- diskcomp has "zero mandatory dependencies" as core feature (single .py file generation)
- All Phase 07.1 features implementable with stdout + ANSI codes
- Both Rich and ANSI implementations required (no Rich-only screens)

## Common Pitfalls

### Pitfall 1: Numbered Menu Works on macOS/Linux but Fails on Windows cmd.exe

**What goes wrong:** Cursor positioning codes (e.g., `\033[2J` to clear screen) fail silently or render garbage on Windows cmd.exe before Windows 10 version 1511. Developer doesn't notice because they test on macOS.

**Why it happens:** Windows cmd.exe historically did not support ANSI codes until 2016. Some enterprise environments still use Windows 7 or unpatched Windows 10.

**How to avoid:**
1. Test on Windows 10+ cmd.exe, PowerShell, and Windows Terminal (diskcomp already supports these)
2. Avoid cursor control—use `print()` and `input()` only (no ANSI clear/cursor-move)
3. If cursor control needed, use Rich (which handles Windows compatibility) or gate code with `os.name == 'nt'` + explicit terminal.enable_vt_processing()

**Warning signs:** User reports "menu numbers show, but input prompt doesn't render" on Windows cmd.exe.

### Pitfall 2: Empty Input Accepted as No-Op (Silent Failure)

**What goes wrong:** User presses Enter without typing anything; `input()` returns `""`, code checks `if response == 'y'`, evaluates False, proceeds with default action without confirming intent.

**Why it happens:** Developers often write `response = input(...); if response == 'y': delete()` without validating that response is non-empty.

**How to avoid:**
1. Always wrap input in a constrained validation function that loops until valid
2. Never accept empty input as "default" in deletion/skip flows
3. Test by pressing Enter without typing—should re-prompt, not proceed

**Warning signs:** User deletes files by accident because they pressed Enter instead of 'y'. Undo log is the only salvation.

### Pitfall 3: Stateful Navigation Loses Prior Selections

**What goes wrong:** User selects folders to skip, then navigates "back" to main menu, then re-enters the folder skip screen. Their previous selections are gone.

**Why it happens:** Folder skip screen reconstructs the list from scratch instead of loading from NavigationContext. Re-hashing or re-reading results loses the prior state.

**How to avoid:**
1. Pass NavigationContext through all screen functions
2. On entry, check if `context.selected_folders_skip` is already populated
3. If populated, display them as pre-selected and allow user to modify
4. Only re-populate if context is empty (first visit)

**Warning signs:** User makes careful selection of 50 folders to skip, goes back to check something else, and folders are forgotten.

### Pitfall 4: Multi-Select Library Adds Mandatory Dependency

**What goes wrong:** Dev adds `questionary` to make multi-select "nice", but now single .py build requires questionary installed. Breaks the "zero mandatory deps" promise.

**Why it happens:** Easy to forget that diskcomp has two distribution paths: pip (can have optional deps) and single .py file (cannot pull extra deps). Not documenting the constraint.

**How to avoid:**
1. Any new multi-select must be optional or hand-rolled using only stdlib + input()
2. If adding a library, it MUST be in pyproject.toml's `[project.optional-dependencies]`, not `[project.dependencies]`
3. Test single .py build before shipping: `python3 diskcomp.py --help` must work without questionary installed

**Warning signs:** Single .py build fails with ImportError for questionary.

### Pitfall 5: Back Navigation Changes Interpretation of "Delete"

**What goes wrong:** User in batch mode reviews deletion preview, goes back to skip some files, then starts deletion again. But the second deletion run's "dry-run preview" now shows different candidates because skipped files were filtered. User is confused.

**Why it happens:** Not resetting the "step counter" or not communicating which step the user is in after back navigation.

**How to avoid:**
1. After back navigation, clearly display which operation is about to happen
2. Show updated candidate count: "Ready to delete {N} files (after skipping {M} folders, {K} flagged)"
3. Never change the semantics of an operation after back navigation—if preview showed 100 files and user goes back to flag some, preview should now show 95 files

**Warning signs:** User reports "I went back and now deletion shows different numbers." Undo log might not match what was displayed.

### Numbered List + Comma Input Parser (Reusable)
```python
# Source: Custom pattern, should live in diskcomp/cli.py

def parse_selection_input(input_str: str, max_index: int) -> List[int]:
    """
    Parse user input like "1,3,5" or "1-3,5" into sorted list of indices.

    Args:
        input_str: User's input (e.g., "1,3,5" or "1-3")
        max_index: Maximum valid index (1-based; e.g., 10 means 1-10 are valid)

    Returns:
        Sorted list of unique indices (1-based): [1, 3, 5]

    Raises:
        ValueError: If format invalid or indices out of range
    """
    if not input_str.strip():
        raise ValueError("Empty input. Please enter at least one number.")

    # Normalize input: handle spaces, tabs, mixed separators
    input_str = input_str.strip().replace('\t', ' ')

    indices = set()

    # Split by comma, then by space
    parts = [p.strip() for p in input_str.split(',')]
    for part in parts:
        if not part:
            continue

        # Check for range (e.g., "1-3" means 1, 2, 3)
        if '-' in part:
            range_parts = part.split('-')
            if len(range_parts) != 2:
                raise ValueError(f"Invalid range format '{part}'. Use 'start-end' (e.g., 1-3).")

            try:
                start, end = int(range_parts[0].strip()), int(range_parts[1].strip())
            except ValueError:
                raise ValueError(f"Invalid range '{part}': not integers.")

            if start < 1 or end > max_index:
                raise ValueError(f"Range {start}-{end} out of bounds (valid: 1-{max_index}).")
            if start > end:
                raise ValueError(f"Range {start}-{end}: start > end.")

            indices.update(range(start, end + 1))
        else:
            # Single index
            try:
                idx = int(part.strip())
            except ValueError:
                raise ValueError(f"Invalid index '{part}': not a number.")

            if idx < 1 or idx > max_index:
                raise ValueError(f"Index {idx} out of bounds (valid: 1-{max_index}).")

            indices.add(idx)

    if not indices:
        raise ValueError("No valid selections made.")

    return sorted(list(indices))

# Test cases:
# "1,3,5" -> [1, 3, 5]
# "1-3" -> [1, 2, 3]
# "1-3,5,7-9" -> [1, 2, 3, 5, 7, 8, 9]
# "" -> ValueError("Empty input")
# "1,10" with max_index=5 -> ValueError("Index 10 out of bounds")
```

### Navigation Context (Preserve State Across Screens)
```python
# Source: Custom pattern, should live in diskcomp/types.py

from dataclasses import dataclass, field
from typing import Set, Optional, List

@dataclass
class NavigationContext:
    """
    Preserve user selections and scan results across back navigation.

    Allows user to go back from folder skip → file flagging → deletion preview,
    then come back without losing their previous selections.
    """
    scan_results: Optional['DuplicateResults'] = None
    keep_path: Optional[str] = None
    other_path: Optional[str] = None
    report_path: Optional[str] = None

    # User selections (accumulate across screens, cleared only by user action)
    selected_folders_skip: Set[str] = field(default_factory=set)
    flagged_files: Set[str] = field(default_factory=set)  # Files marked "do not delete"

    def reset_selections(self):
        """Clear all user selections (for explicit 'restart' action)."""
        self.selected_folders_skip.clear()
        self.flagged_files.clear()

    def get_candidates_after_filtering(self, original_candidates: List['DuplicateFile']) -> List['DuplicateFile']:
        """
        Return candidates after applying folder skip and file flagging filters.

        Args:
            original_candidates: Full list from classification

        Returns:
            Filtered list (excluding skipped folders and flagged files)
        """
        filtered = original_candidates

        # Filter 1: Remove files in skipped folders
        if self.selected_folders_skip:
            filtered = [f for f in filtered if not any(
                f.path.startswith(folder_path) for folder_path in self.selected_folders_skip
            )]

        # Filter 2: Remove flagged files
        if self.flagged_files:
            filtered = [f for f in filtered if f.path not in self.flagged_files]

        return filtered
```

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Bare `input()` without validation | `prompt_confirm()` loop with constrained keys | Phase 07.1 | Eliminates silent acceptance of empty/invalid input |
| No multi-select (single choice per screen) | Numbered lists + comma input | Phase 07.1 | Allows users to skip multiple folders in one operation |

- InquirerPy / Questionary: Added unnecessary dependencies for numbered menu functionality that hand-rolling solves in ~30 lines
- State machine libraries (python-statemachine): Overkill for 5-screen workflow; while loops + context object sufficient

## Open Questions

1. **Should folder skip be pre-hashing or post-hashing?**
   - **Current decision (Phase 07.1 spec):** Post-hashing (after candidates identified, before deletion)
   - **Alternative:** Pre-scan (let user pick folders before size filter runs)
   - **Trade-off:** Pre-scan = more user control, but risk of re-hashing if user goes back. Post-hashing = faster UX (hash is done), but less granular control
   - **Recommendation:** Stick with post-hashing (spec). If user needs pre-scan control, that's Phase 08+ feature.

2. **For file flagging, should UI show file size/hash/mod date in the list?**
   - **Minimal:** Just path: `1) /Users/wilks/Documents/backup.zip`
   - **Enhanced:** `1) /Users/wilks/Documents/backup.zip (285 MB)`
   - **Detailed:** `1) /Users/wilks/Documents/backup.zip (285 MB) | hash: abc123... | mod: 2024-01-15`
   - **Trade-off:** More info = better decisions, but list becomes wide/long
   - **Recommendation:** Include size in paren: `path (size)` — balances UX and info density

3. **What if user presses Ctrl+C during folder skip / file flagging selection?**
   - **Current approach (implicit):** Let KeyboardInterrupt propagate, write undo log if deletion was partially started
   - **Alternative:** Catch, ask "are you sure?" with y/n confirmation
   - **Recommendation:** Let it propagate. These are pre-deletion screens (no files deleted yet), so Ctrl+C is safe. Only protect actual deletion runs.

4. **Should back navigation be available on deletion confirmation ("Type DELETE to confirm")?**
   - **Option A:** Yes, 'b' returns to menu, doesn't delete
   - **Option B:** No, once in batch deletion flow, no going back (must Ctrl+C)
   - **Recommendation:** Yes, allow 'b' to abort gracefully. Users may change mind at last moment.
